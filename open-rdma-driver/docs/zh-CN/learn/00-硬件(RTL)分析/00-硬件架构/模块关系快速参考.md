## 模块职责速查表

| 模块名称 | 中文名称 | 核心职责 | 输入 | 输出 |
|---------|---------|---------|------|------|
| **BsvTopOnlyHardIp** | 硬件IP封装 | PCIe和以太网硬件接口封装 | PCIe/Ethernet 信号 | 数据流接口 |
| **CsrRootConnector** | CSR根连接器 | 将PCIe内存访问转为寄存器访问 | PCIe写请求 | CSR寄存器操作 |
| **RingbufAndDescriptorHandler** | 环形缓冲区处理器 | 管理软硬件交互队列 | 主存描述符 | WQE、完成通知 |
| **QpMrPgtQpc** | RDMA核心处理 | QP/MR/页表管理，协议处理 | WQE | 以太网帧 |
| **SQ** | 发送队列 | 处理发送请求 | WorkQueueElem | RDMA数据包 |
| **RQ** | 接收队列 | 处理接收数据 | 以太网帧 | 完成描述符 |
| **PayloadGen** | Payload生成器 | 从内存读取发送数据 | PayloadGenReq | 数据流 |
| **PayloadCon** | Payload消费器 | 将接收数据写入内存 | 数据流 | DMA写 |
| **PacketGen** | 数据包生成器 | 组装RDMA数据包 | WQE + Payload | 以太网帧 |
| **PacketParse** | 数据包解析器 | 解析RDMA数据包 | 以太网帧 | 元数据 + Payload |
| **QPContext** | QP上下文 | 存储QP状态信息 | QP查询 | QP状态 |
| **MRTable** | 内存区域表 | 管理注册的内存区域 | lkey/rkey | 页表信息 |
| **AddressTranslate** | 地址转换器 | 虚拟地址到物理地址转换 | 虚拟地址 | 物理地址 |
| **TopLevelDmaChannelMux** | DMA通道复用器 | 多个DMA源的仲裁和复用 | 多路DMA请求 | PCIe DMA |
| **AutoAckGenerator** | 自动ACK生成器 | 生成ACK/NAK数据包 | 接收状态 | ACK数据包 |

## 数据结构速查表

| 数据结构 | 用途 | 关键字段 | 流转路径 |
|---------|------|---------|---------|
| **WorkQueueElem** | 工作队列元素 | opcode, laddr, raddr, len, psn | Ringbuf → SQ → PacketGen |

| **RETH** | 远程以太网传输头部 | va, rkey, dlen | RDMA Write/Read使用 |

## 关键接口速查表

### Pipe 接口（最常用的数据传输接口）

```bsv
interface PipeOut#(type t);
    method t first();      // 查看队首元素
    method Action deq();   // 出队
    method Bool notEmpty;  // 是否非空
endinterface

interface PipeIn#(type t);
    method Action enq(t x);  // 入队
    method Bool notFull;     // 是否未满
endinterface

interface PipeInB0#(type t);  // B0 = Bypass 0级，更低延迟
    method Action enq(t x);
    method Bool notFull;
endinterface
```

### Client/Server 接口（请求-响应模式）

```bsv
interface ClientP#(type req_t, type resp_t);
    method ActionValue#(req_t) getReq();   // 发送请求
    method Action putResp(resp_t resp);     // 接收响应
endinterface

interface ServerP#(type req_t, type resp_t);
    method Action putReq(req_t req);       // 接收请求
    method ActionValue#(resp_t) getResp(); // 发送响应
endinterface
```

## 发送路径快速查找

```
用户 ibv_post_send
    ↓
主存 WQE Ringbuf
    ↓ (DMA读)
WQE Ringbuf 模块
    ↓ (解析)
WorkQueueDescParser
    ↓ WorkQueueElem
SQ (发送队列)
    ├─→ 查询 QP Context
    ├─→ 查询 MR Table
    └─→ 生成 PayloadGenReq
        ↓
PayloadGen
    ├─→ 地址分块
    ├─→ 地址转换 (查询 Page Table)
    └─→ DMA 读数据
        ↓ DataStream

    ├─→ 按 PMTU 分包
    ├─→ 生成 RDMA 头部 (BTH, RETH等)
    └─→ 对齐和分割 Payload
        ↓ RDMA Packet
EthernetPacketGen
    ├─→ 添加 UDP 头部
    ├─→ 添加 IP 头部
    └─→ 添加 Ethernet 头部
        ↓ Ethernet Frame
FTileMac (以太网 MAC)
    ↓
网络
```

## 接收路径快速查找

```
网络
    ↓ Ethernet Frame
FTileMac (以太网 MAC)
    ↓
PacketParse
    ├─→ 解析 Ethernet/IP/UDP 头部
    ├─→ 解析 RDMA 头部 (BTH + 扩展)
    ├─→ 提取 RDMA 元数据
    └─→ 提取 Payload DataStream
        ↓
RQ (接收队列)
    ├─→ 查询 QP Context (校验 PSN)
    ├─→ 查询 MR Table (校验 rkey)
    ├─→ 确定写入地址
    └─→ 生成 PayloadConReq
        ↓
PayloadCon
    ├─→ 地址分块
    ├─→ 地址转换 (查询 Page Table)
    ├─→ 数据流分割
    └─→ DMA 写数据
        ↓
主存 (用户数据写入完成)
        ↓
RQ 生成完成描述符
        ↓
CQ Ringbuf
        ↓ (DMA写)
主存 CQ Ringbuf
        ↓
用户 ibv_poll_cq
```

## CSR 寄存器地址空间

```
CSR 地址空间布局：

0x0000 - 0x0FFF: QP 相关寄存器
    每个 QP 占用 256 字节 (CSR_ADDR_BLOCK_SIZE_FOR_EACH_QP)

    QP0: 0x0000 - 0x00FF
        - WQE Ringbuf Base Addr Low/High
        - WQE Ringbuf Head/Tail
        - Recv Ringbuf Base Addr Low/High
        - Recv Ringbuf Head/Tail

    QP1: 0x0100 - 0x01FF
    QP2: 0x0200 - 0x02FF
    QP3: 0x0300 - 0x03FF

0x1000 - 0x1FFF: Command Queue 寄存器
    - CMD Req Queue Base/Head/Tail
    - CMD Resp Queue Base/Head/Tail

0x2000 - 0x2FFF: Simple NIC 寄存器
    - SimpleNIC TX Queue Base/Head/Tail
    - SimpleNIC RX Queue Base/Head/Tail

0x3000+: QP/MR/PGT 配置寄存器
```

## 常见问题速查

### Q1: 为什么有 4 个 SQ 和 RQ？
**A**: 支持 4 个独立的 QP 并行工作，提高吞吐量。每个 QP 对应一个虚拟通道。

### Q2: PayloadGen 和 PacketGen 有什么区别？
**A**:
- **PayloadGen**: 从内存读取**用户数据**（Payload）
- **PacketGen**: 生成**RDMA协议头部**并组装成完整数据包

### Q3: 地址转换为什么需要 8 路并发？
**A**:
- 每个 QP 的发送和接收各需要 1 路
- 4 个 QP × 2（发送+接收）= 8 路

### Q4: DMA 通道复用的目的是什么？
**A**:
- PCIe 接口的通道数有限（如 4 个）
- 多个模块（Ringbuf、Payload、PGT等）需要共享这些通道
- 通过仲裁器实现公平访问

### Q5: 为什么需要 Ringbuf？
**A**:
- 软硬件之间的异步通信机制
- 软件写描述符，硬件轮询读取
- 硬件写完成，软件轮询读取
- 无需中断，高效率
