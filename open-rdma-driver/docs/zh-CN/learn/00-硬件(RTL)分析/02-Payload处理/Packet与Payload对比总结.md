# Packet 与 Payload 对比总结

## 核心概念对比

### Packet（数据包）

**定义**：完整的网络传输单元，包含所有协议层的信息。

**组成结构**：
```
┌─────────┬────────┬────────┬──────────────────────┬──────────┬─────┐
│以太网头部│ IP头部 │UDP头部 │ RDMA头部(BTH+扩展头部) │ Payload  │ CRC │
└─────────┴────────┴────────┴──────────────────────┴──────────┴─────┘
 \                                                    /
  \_______________  Packet 的范围  ________________/
```

**关键特征**：
- **完整性**：包含从物理层到传输层的所有必要信息
- **大小限制**：受 PMTU（Path MTU）限制，典型值为 256/512/1024/2048/4096 字节
- **独立性**：每个 packet 可以独立路由和传输
- **可见性**：网络设备（路由器、交换机）可以查看和处理

**类比**：快递包裹（包括外包装盒、地址标签、物流单等完整信息）

---

### Payload（有效负载）

**定义**：数据包中实际的用户数据部分，不包含任何协议头部。

**位置**：
```
┌─────────┬────────┬────────┬──────────────────────┬──────────┬─────┐
│以太网头部│ IP头部 │UDP头部 │ RDMA头部(BTH+扩展头部) │ Payload  │ CRC │
└─────────┴────────┴────────┴──────────────────────┴──────────┴─────┘
                                                     \________/
                                                    Payload 的范围
```

**关键特征**：
- **纯数据**：只包含用户关心的实际数据
- **可变长**：长度取决于用户需求和 PMTU 限制
- **不可路由**：没有地址信息，无法单独传输
- **业务相关**：内容和格式由应用程序决定

**类比**：包裹里的实际物品（不包括包装）

---

## 详细对比表

| 维度 | Packet（数据包） | Payload（有效负载） |
|------|-----------------|-------------------|
| **定义** | 完整的网络传输单元 | 数据包中的用户数据部分 |
| **组成** | 头部 + Payload + 校验和 | 纯用户数据 |
| **大小** | 受 PMTU 限制（如 4096 字节） | 可变，受 Packet 大小限制 |
| **作用** | 网络传输的基本单位 | 实际要传输的内容 |
| **处理位置** | PacketGenAndParse.bsv | PayloadGenAndCon.bsv |
| **关注点** | 协议、路由、可靠性 | 数据本身的存储和读写 |
| **可见性** | 网络设备可见 | 仅端点可见 |
| **独立性** | 可独立传输 | 必须封装在 Packet 中 |

## 层次关系

### 概念层次

```
┌──────────────────────────────────────┐
│          应用层数据 (Message)          │  ← 用户视角：一条完整的消息
└─────────────────┬────────────────────┘
                  │
          (可能需要分片，如果超过 PMTU)
                  │
                  ▼
    ┌─────────────────────────────────┐
    │        Payload (有效负载)         │  ← Payload 视角：消息的一部分
    └─────────────────┬───────────────┘
                      │
            (添加 RDMA/UDP/IP/Ethernet 头部)
                      │
                      ▼
        ┌─────────────────────────────┐
        │      Packet (数据包)         │  ← Packet 视角：可传输的单元
        └──────────────────────────────┘
```

### 处理层次

```
发送端：

┌─────────────┐
│ 用户内存数据 │
└──────┬──────┘
       │
       ▼
┌─────────────────────┐
│  PayloadGen         │  ← 从内存读取 Payload
│  (PayloadGenAndCon) │
└──────┬──────────────┘
       │ Payload 数据流
       ▼
┌─────────────────────┐
│  PacketGen          │  ← 将 Payload 封装成 Packet
│ (PacketGenAndParse) │
└──────┬──────────────┘
       │ 完整 Packet
       ▼
┌─────────────┐
│  网络发送    │
└─────────────┘


接收端：

┌─────────────┐
│  网络接收    │
└──────┬──────┘
       │ 完整 Packet
       ▼
┌─────────────────────┐
│  PacketParse        │  ← 解析 Packet，提取 Payload
│ (PacketGenAndParse) │
└──────┬──────────────┘
       │ Payload 数据流
       ▼
┌─────────────────────┐
│  PayloadCon         │  ← 将 Payload 写入内存
│  (PayloadGenAndCon) │
└──────┬──────────────┘
       │
       ▼
┌─────────────┐
│ 用户内存数据 │
└─────────────┘
```

## 文件职责对比

### PacketGenAndParse.bsv

**主要职责**：Packet 层的处理

| 模块 | 功能 |
|------|------|
| **PacketGen** | • 根据工作队列元素生成 RDMA 头部<br>• 计算数据包大小和数量<br>• 分配 PSN（数据包序列号）<br>• 生成 MAC/IP/UDP 元数据<br>• 按 PMTU 分包 |
| **PacketParse** | • 解析以太网帧<br>• 提取 RDMA 头部信息<br>• 分离 Payload 和控制信息<br>• 数据包分类（RDMA vs 其他） |
| **PacketGenReqArbiter** | • 多 QP 发送请求仲裁<br>• 公平调度和带宽分配 |

**关注的问题**：
- 如何将大消息分成多个 Packet？
- 每个 Packet 应该包含哪些头部？
- 如何确保 Packet 的顺序和完整性？
- 如何处理不同类型的 RDMA 操作（WRITE, READ, SEND, ATOMIC）？

---

### PayloadGenAndCon.bsv

**主要职责**：Payload 层的处理

| 模块 | 功能 |
|------|------|
| **PayloadGen** | • 从用户内存读取数据<br>• 虚拟地址到物理地址转换<br>• DMA 读取<br>• 数据流拼接 |
| **PayloadCon** | • 接收 Payload 数据流<br>• 虚拟地址到物理地址转换<br>• DMA 写入<br>• 数据流分割 |

**关注的问题**：
- 如何高效地从内存读取/写入数据？
- 如何处理虚拟地址到物理地址的转换？
- 如何确保数据对齐？
- 如何优化 DMA 传输效率？

## 协同工作示例

### 场景：RDMA Write 操作（发送 10KB 数据）

**假设**：
- PMTU = 4096 字节
- RDMA 头部 = 56 字节（BTH + RETH）
- 可用 Payload 空间 = 4096 - 56 = 4040 字节/包

#### 步骤1：Payload 准备（PayloadGen）

```
用户请求：写 10KB 数据，从本地地址 0x7fff1000

PayloadGen 处理：
1. 查询内存区域表，获取页表信息
2. 将 10KB 按 PCIe 突发大小（128B）分块：
   - 块1: 128B (0x7fff1000-0x7fff107f)
   - 块2: 128B (0x7fff1080-0x7fff10ff)
   - ...
   - 块80: 128B

3. 对每块进行地址转换：
   VA 0x7fff1000 → PA 0x80001000
   VA 0x7fff1080 → PA 0x80001080
   ...

4. 发起 DMA 读请求，获取数据

5. 拼接成连续的 10KB 数据流
```

#### 步骤2：Packet 生成（PacketGen）

```
PacketGen 处理：
1. 接收 PayloadGenReq：
   - 本地地址：0x7fff1000
   - 远程地址：0x8000a000
   - 长度：10240 字节
   - PMTU：4096 字节

2. 计算分包策略：
   需要 3 个 Packet
   - Packet 1: Payload = 4040B, 类型 = RDMA_WRITE_FIRST
   - Packet 2: Payload = 4040B, 类型 = RDMA_WRITE_MIDDLE
   - Packet 3: Payload = 2160B, 类型 = RDMA_WRITE_LAST

3. 为每个 Packet 生成头部：

   Packet 1:
   ┌──────────────────────────────────────────┐
   │ BTH: opcode=RDMA_WRITE_FIRST, PSN=100    │
   │ RETH: raddr=0x8000a000, rkey=0x123,      │
   │       dlen=10240                         │
   ├──────────────────────────────────────────┤
   │ Payload: 4040 字节 (偏移 0-4039)         │
   └──────────────────────────────────────────┘

   Packet 2:
   ┌──────────────────────────────────────────┐
   │ BTH: opcode=RDMA_WRITE_MIDDLE, PSN=101   │
   ├──────────────────────────────────────────┤
   │ Payload: 4040 字节 (偏移 4040-8079)      │
   └──────────────────────────────────────────┘

   Packet 3:
   ┌──────────────────────────────────────────┐
   │ BTH: opcode=RDMA_WRITE_LAST, PSN=102     │
   ├──────────────────────────────────────────┤
   │ Payload: 2160 字节 (偏移 8080-10239)     │
   └──────────────────────────────────────────┘

4. 添加 UDP/IP/Ethernet 头部

5. 输出到网络接口
```

#### 步骤3：Packet 接收（PacketParse）

```
接收端 PacketParse 处理：
1. 接收 Ethernet 帧

2. 检查目标 MAC、IP、UDP 端口
   → 确认是 RDMA 数据包

3. 解析 BTH：
   - opcode = RDMA_WRITE_FIRST/MIDDLE/LAST
   - PSN = 100/101/102
   - dqpn = 目标 QP 号

4. 解析扩展头部（如果是 FIRST）：
   - RETH: raddr=0x8000a000, rkey=0x123, dlen=10240

5. 提取 Payload：
   - Packet 1 → 4040 字节
   - Packet 2 → 4040 字节
   - Packet 3 → 2160 字节

6. 输出 Payload 数据流和元数据
```

#### 步骤4：Payload 写入（PayloadCon）

```
PayloadCon 处理：
1. 接收 PayloadConReq：
   - 目标地址：0x8000a000
   - 长度：10240 字节

2. 按 PCIe 突发大小分块：
   - 块1: 128B (0x8000a000-0x8000a07f)
   - 块2: 128B (0x8000a080-0x8000a0ff)
   - ...

3. 地址转换：
   VA 0x8000a000 → PA 0x90001000
   VA 0x8000a080 → PA 0x90001080
   ...

4. 将接收的 Payload 数据流分割：
   - 数据片段1: 128B → DMA 写 PA 0x90001000
   - 数据片段2: 128B → DMA 写 PA 0x90001080
   - ...

5. 全部完成后，输出 True（完成信号）
```

## 关键设计原则

### 1. 职责分离

```
PayloadGenAndCon（Payload 层）：
  ✓ 关心：数据在哪里（地址）
  ✓ 关心：如何读写数据（DMA）
  ✗ 不关心：数据如何传输（网络协议）

PacketGenAndParse（Packet 层）：
  ✓ 关心：数据如何封装（协议头部）
  ✓ 关心：数据如何分包（PMTU）
  ✗ 不关心：数据从哪里来（内存管理）
```

### 2. 接口清晰

```bsv
// Payload → Packet 接口（发送）
interface PipeOut#(DataStream) payloadGenStreamPipeOut;  // PayloadGen 输出
interface PipeIn#(DataStream) genRespPipeIn;             // PacketGen 输入

// Packet → Payload 接口（接收）
interface PipeOut#(DataStream) rdmaPayloadPipeOut;       // PacketParse 输出
interface PipeIn#(DataStream) payloadConStreamPipeIn;    // PayloadCon 输入
```

### 3. 流水线化

两个文件都采用深度流水线设计：

```
PayloadGen:
  handleInReq → 地址转换 → DMA读 → 数据拼接
  (每个阶段独立，可并行处理多个请求)

PacketGen:
  查询MR → 分块 → 生成头部1 → 生成头部2
  (4级流水线，吞吐量 = 每周期1个Packet)
```

### 4. 错误隔离

```
Payload 层错误：
  - 地址转换失败
  - 内存保护违规
  - DMA 错误
  → 影响单个内存操作

Packet 层错误：
  - 无效操作码
  - PSN 错误
  - PMTU 超限
  → 影响单个数据包
```

## 性能优化协同

### 1. 地址对齐

**问题**：本地地址和远程地址可能不对齐

**解决方案**：
- PayloadGen 读取时按**本地地址**对齐
- PacketGen 中的 StreamShifter 进行对齐转换
- 确保 Payload 按**远程地址**对齐

```
本地：[--AB|CDEF|GHIJ]  (偏移2)
      ↓ StreamShifter (右移2)
网络：[----A|BCDE|FGHI|J---]  (偏移4，远程对齐)
```

### 2. 批量处理

**PayloadGen**：按 PCIe 突发大小（128B）批量读取
**PacketGen**：按 PMTU 大小（4KB）批量发送

```
128B 突发 × 32 = 4KB Packet
(提高 PCIe 和网络的利用率)
```

### 3. 并行操作

```
时间轴：
t1: PayloadGen 读块1 | PacketGen 生成Packet N
t2: PayloadGen 读块2 | PacketGen 生成Packet N+1
t3: PayloadGen 读块3 | PacketGen 生成Packet N+2
...
(两个模块独立流水，互不阻塞)
```

## 总结

### Packet vs Payload 的本质区别

| 方面 | Packet | Payload |
|------|--------|---------|
| **抽象层次** | 网络传输层 | 数据存储层 |
| **关注点** | 如何在网络中传输 | 如何在内存中存取 |
| **生命周期** | 从发送到接收 | 从读到写 |
| **可见性** | 网络设备可见 | 仅端点可见 |
| **处理复杂度** | 协议栈复杂 | 内存管理复杂 |

### 两个文件的协作模式

```
PacketGenAndParse.bsv  ←→  PayloadGenAndCon.bsv
     (Packet 层)              (Payload 层)
         ↓                         ↓
    "打包/拆包"              "装货/卸货"
         ↓                         ↓
   关心如何传输            关心数据在哪里
```

**类比快递系统**：
- **PayloadGenAndCon** = 仓库（货物进出）
- **PacketGenAndParse** = 快递公司（打包和配送）

### 实现 RDMA 零拷贝的关键

```
传统 I/O：
  应用内存 → [CPU拷贝] → 内核缓冲区 → [CPU拷贝] → 网卡 → 网络

RDMA (这两个文件的协作)：
  应用内存 → [PayloadGen: DMA读] → [PacketGen: 添加头部] → 网卡 → 网络
            ↑__________________ 无 CPU 拷贝 __________________↑
```

**零拷贝的实现**：
1. PayloadGen 通过 DMA 直接读取用户内存（无 CPU 拷贝）
2. PacketGen 添加头部时使用硬件流水线（无 CPU 参与）
3. PayloadCon 通过 DMA 直接写入用户内存（无 CPU 拷贝）

**结果**：
- 延迟：微秒级（vs 传统 I/O 的毫秒级）
- CPU 占用：接近 0%（vs 传统 I/O 的 50-100%）
- 带宽：接近网络线速（100Gbps+）
