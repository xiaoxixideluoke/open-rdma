# Bitmap窗口参数详解：112、128和leftBound的特殊含义

## 为什么是128位bitmap？

### 1. 硬件设计考量
- **CPU友好**：128位 = 2×64位，正好对应现代CPU的寄存器宽度
- **并行处理**：一次可以处理128个包的状态，提高吞吐量
- **内存对齐**：128位自然对齐，访问效率高

### 2. 协议设计权衡
- **窗口大小**：128个包 ≈ 典型RDMA流水线的深度
- **内存占用**：每QP仅16字节，支持大量并发QP
- **延迟覆盖**：128包 × MTU ≈ 足够覆盖网络往返延迟

## 112的特殊含义

### 数学关系
```
112 = 128 (bitmap宽度) - 16 (stride偏移)
```

### 为什么硬件从stride 16开始？

1. **预填充缓冲区**：硬件提前分配16个包的空间
2. **流水线优化**：允许硬件提前准备数据，隐藏延迟
3. **边界保护**：防止PSN 0的特殊处理

### PSN重映射的实际意义
```
软件PSN = 硬件PSN - 112
```

| 硬件PSN | 软件PSN | 含义 |
|---------|---------|------|
| 0-15    | 65424-65439 | wrap-around区域 |
| 16-127  | 0-111   | 正常窗口前半部分 |
| 128-255 | 16-143  | 正常窗口后半部分 |

## leftBound的作用

### 定义
`leftBound`是当前bitmap窗口的左边界，表示窗口中第一个bit对应的PSN。

### 更新条件
从`PsnContinousChecker.bsv`代码：
```bsv
// 当新PSN超出当前窗口8个stride时滑动窗口
isShiftOutOfBoundary = (boundaryDelta >= 8)
```

其中：
- `8 = ACK_BITMAP_WIDTH / ACK_WINDOW_STRIDE = 128 / 16`
- 窗口提前滑动，确保有足够的缓冲空间

### 滑动过程
```
初始：leftBound = 16, bitmap = [16..143]
滑动后：leftBound = 32, bitmap = [32..159]
```

## 参数间的协同工作

### 1. 包到达处理
```
1. 计算相对位置：pos = (psn - leftBound) % 128
2. 设置bitmap：bitmap |= (1 << pos)
3. 检查连续性：连续1的个数
```

### 2. 窗口滑动决策
```
1. 找到最右侧连续1的边界
2. 如果边界 > leftBound + 8，滑动窗口
3. 新leftBound = 边界位置
4. 清空旧bitmap
```

### 3. 硬件-软件同步
```
硬件侧：
- 维护leftBound和bitmap
- 生成ACK时附带当前PSN和bitmap

软件侧：
- 接收重映射的PSN (psn - 112)
- 更新本地跟踪状态
- 处理完成和重传
```

## 设计哲学

这些看似随意的数字（128、112、16）实际上体现了精心的系统设计：

1. **性能与资源的平衡**：128位提供了良好的并行度，同时保持内存开销合理
2. **流水线优化**：16的偏移为硬件提供了预处理和缓冲的空间
3. **鲁棒性设计**：提前滑动窗口（+8）确保不会丢失确认信息
4. **硬件友好**：所有参数都便于硬件高效实现

## 实际影响

理解这些参数有助于：
- 调试PSN相关的问题
- 优化RDMA性能
- 理解硬件行为的预期
- 正确解释日志中的PSN值

例如，日志中PSN=0实际上表示硬件PSN=112，这是完全正常的行为，不代表错误。
