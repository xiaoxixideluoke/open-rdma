## ACK Timeout 和 Retry 机制问题 低优先级

**当前问题**：modify_qp verbs 中传入的 timeout 和 retry_cnt 参数被完全忽略

**现状分析**：
1. ACK timeout 参数在设备初始化时从配置文件或常量设置，全局应用于所有 QP
2. 虽然 `IbvQpAttr` 结构体定义了 `timeout` 和 `retry_cnt` 字段（src/rdma_utils/types.rs:440-441）
3. 但在 `update_qp()` 函数中（src/verbs/ctx.rs:342-403），这些参数被完全忽略
4. 当前 timeout 配置方式：
   - 编译时常量：src/constants.rs:36-39
     - DEFAULT_INIT_RETRY_COUNT = 5
     - DEFAULT_TIMEOUT_CHECK_DURATION = 8
     - DEFAULT_LOCAL_ACK_TIMEOUT = 50（4.096μs × 2^50 = 4.5天，过大！）
   - 运行时配置：/etc/bluerdma/config.toml（如果存在）
5. Timeout 在设备初始化时加载（src/verbs/core.rs:60），传递给 QpAckTimeoutWorker

**需要改进**：
1. 修改 `update_qp()` 函数，使其能够处理 per-QP 的 timeout 和 retry_cnt 设置
2. 调整默认的 DEFAULT_LOCAL_ACK_TIMEOUT 值（50 太大，建议使用更合理的值如 14-18）
3. 设计 per-QP timeout 管理机制：
   - 在 QpAckTimeoutWorker 中支持动态更新单个 QP 的 timeout 配置
   - 或者在 TransportTimer 创建后支持参数更新
4. 确保符合 RDMA 规范：应用程序通过 modify_qp 设置的 timeout 参数应该生效
